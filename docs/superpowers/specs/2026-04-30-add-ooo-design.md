# Add Oscuro's Oblivion Overhaul (OOO) to Reborn — Design

**Status:** Ready for review
**Date:** 2026-04-30
**Project:** APW Reborn — Oblivion modlist
**Approach:** B (OOO + MOO combo, only patches for mods Reborn currently loads, defer worldspace overhauls)

---

## Goal

Add OOO Updated 1.7.0.0 to Reborn-Base alongside the existing Maskar's Oblivion Overhaul (MOO), reaching feature parity with APW's OOO+MOO setup but without the BC/UL/Kvatch Rebuilt/MMM dependencies APW carried. End state: Reborn ships an OOO+MOO combo, conflicts resolved with documented provenance, all changes promoted into `Reborn-Base` and packaged via the existing Wabbajack pipeline.

## Why now

Reborn currently ships only MOO as an overhaul. OOO adds substantially more leveled-list/loot/creature/quest content (~4500 NPC_+CREA records, ~232 LVLI/LVLC, ~408 CONT, ~131 SCPT records that APW had to resolve). MOO's stock `compatibility.ini` already auto-detects OOO and reconfigures itself when the ESM is loaded — meaning OOO+MOO is a designed-for combo, not a hack. The infrastructure to integrate cleanly (`oblivion-conflicts`, `oblivion-batch-compile`, Bash-mergeable workflow, Reborn Configs override pattern, manifest-driven status tracking) is all in place from prior phases.

## Scope decisions made during brainstorm

1. **Approach B chosen over A/C:** OOO+MOO combo with patches limited to mods Reborn currently loads. APW's BC/UL/Kvatch Rebuilt/MMM patches are out of scope. (Kvatch Rebuilt may come back later as a sub-project.)
2. **Modules B chosen:** OOO Updated + OOO Unofficial Patch + OOO Shivering Isles. Skip OOO Enhanced Resources (~14k loose files, visual-only).
3. **CR strategy B chosen:** Subset APW CR to records intersecting Reborn's master set AND touching OOO records, then use `oblivion-conflicts -conflicts` for the residue. Strict diff-pattern hashing for the analyzer.
4. **OOOExtended (Nexus 47177) explicitly excluded** even though archives are downloaded — keeps APW parity for content; revisit later if desired.
5. **OOO Add-ons (26 legacy 2008 ESPs) deferred to v2** — all opt-in tweaks, easier to evaluate after the base integration lands.

## Reborn CR ESP construction (analyze-first workflow)

### Why analyze, not blind-copy

APW's `APW - Conflict Resolution.esp` carries 30k+ overrides built against an older OOO version (likely 1.5.x or 1.6.x). Blind-copying records that touch OOO would (a) bake stale resolutions for OOO 1.7's renamed vampires and rewritten boss SCPTs, (b) drag in dependencies on plugins Reborn doesn't have, (c) carry decisions we don't understand. The analyze-first workflow gives us provenance and reproducibility.

### Stage 1 — Analyzer (`analyze_apw_cr_ooo_overrides.pas`)

Pascal script run in xEdit (per `reference_xedit_mo2_launch.md`'s headless-launch pattern). For each APW CR override record where (a) the target master is in Reborn's load order **AND** (b) the record touches OOO (record's owner is OOO, OR any referenced FormID is owned by OOO), emit a row.

Output: `D:\Modlists\Reborn\research\apw_cr_ooo_overrides.csv` (long-form, one row per record) + `apw_cr_ooo_patterns.csv` (one row per unique `diff_pattern_hash` with count + sample records).

Schema:

| field | description |
|---|---|
| `target_fid` | full FormID hex (e.g. `0001A2B3`) |
| `target_master` | the plugin owning the original record (e.g. `Cobl Main.esm`) |
| `signature` | record signature (REFR/LVLI/SCPT/etc.) |
| `editor_id` | EditorID (best-effort) |
| `record_name` | FULL or NAME if available |
| `change_summary` | short prose: what fields changed |
| `diff_pattern_hash` | `sha1` over `(field_path, old_val, new_val)` tuples. **Strict** — same hash means identical edit pattern across all changed fields. |
| `ooo_touch_reason` | "owns record" / "owns N of M referenced FormIDs" / "OOO is winning override absent CR" |
| `target_record_in_current_ooo` | `present` / `changed` / `removed` — flags drift between APW's OOO version and our 1.7.0.0 |
| `recommend` | `merge` / `review` / `drop` heuristic |

`diff_pattern_hash` is the load-bearing trick: hundreds of records often share one resolution pattern (e.g. "merge OOO loot into vanilla list, preserve vanilla min-level"). Group by hash → review N patterns instead of N records.

### Stage 2 — Decision review

Walk `apw_cr_ooo_patterns.csv` (expected 30–80 patterns covering ~2k records) and decide per pattern: **keep / drop / modify**. SCPT/QUST/SPEL/GMST records are reviewed individually — script and quest behavior changes don't pattern-match well, and the record counts are small (131 SCPT, ~3 QUST, 70 SPEL/ENCH).

Decisions go in `D:\Modlists\Reborn\research\cr_decisions.csv`, git-tracked. Schema TBD in implementation plan, but at minimum: `diff_pattern_hash` (or `target_fid` for individual reviews), `decision`, `notes`.

### Stage 3 — Builder (`build_reborn_ooo_cr.pas`)

Pascal script that reads `cr_decisions.csv` and constructs `Reborn - OOO Conflict Resolution.esp` by copying APW CR records as overrides into the new ESP, applying `modify` tweaks where specified, skipping `drop` records. Re-runnable when decisions change.

### SCPT recompile path

Per `feedback_scda_vs_sctx_recompile.md`, xEdit has no SCDA compiler. SCPT records that survive the filter:

- **Default**: copy SCDA blob from APW CR initially (cheap, deterministic — APW's compiled bytecode is known-good).
- **Recompile case**: if the conflict sweep flags a script behavior issue, or `target_record_in_current_ooo=changed` for a SCPT, edit SCTX in xEdit and recompile via `oblivion-batch-compile` (per `reference_oblivion_batch_compile.md`).

### Provenance & re-derivation

- The analyzer is reproducible: re-run if APW CR updates upstream or our master set changes.
- The builder is reproducible: re-run after editing decisions.
- **Never hand-edit `Reborn - OOO Conflict Resolution.esp`** — manual fixes go in a separate `Reborn - Manual CR.esp` so re-runs don't clobber them.

### Loading position

After OOO Updated + OOO Unofficial Patch + OOO Shivering Isles, before Bashed Patch. LOOT will probably want to put it next to other CR ESPs; force position via WB load-order rules if needed.

## MOO settings & OOO detection

MOO ships `Maskar's Oblivion Overhaul for compatibility.ini` that auto-detects `Oscuro's_Oblivion_Overhaul.esp` at load and applies a defined override set. **No manual MOO compat code needed — Maskar wrote it.**

Reborn Configs' `Maskar's Oblivion Overhaul.ini` explicitly sets values that the OOO compat block will silently override at runtime when OOO loads:

| Reborn Configs value | MOO compat overrides to | Effect |
|---|---|---|
| `ini_traps=1` | `0` | OOO traps win |
| `ini_ability_bashlocks=1` | `0` | OOO bash-lock flavor wins |
| `ini_add_magicitems=1` | `0` | OOO populates magic items |
| `ini_update_loot=1` | `0` | OOO leveled lists authoritative |
| `ini_loot_dynamic=1` | `0` | MOO dynamic loot off |
| `ini_bosscontainer_boss/loot=1` | `0` | OOO boss containers win |
| `ini_fDamageWeaponMult=0.66`, `ini_fCreatureCalcDamage=1.33`, etc. | `-1` | revert to OOO/vanilla |
| `ini_fNPCAttributeHealthMult`, `ini_iNPCBasePerLevelHealthMult`, `ini_fDamageToArmorPercentage`, `ini_fDamageToWeaponPercentage` | `-1` | revert |
| `ini_mythicalcreatures_diverse=1` | `0` | OOO handles creature variety |

This is the supported design — running both systems' loot/damage at full intensity double-stacks. **Trust Maskar's compat for v1.**

Reborn's MOO leveling overrides (`levelscaling_npc_overridden_*`, `add_veins=0`, `add_vampires=0`, `ability_heal=0`) are **not** in the OOO compat block and survive untouched.

### Acceptance check

After install: in-game console `getstage MOO` should be `>= 30` (compat block fired), and `show MOO.ini_traps` should be `0` (OOO compat applied). If both pass, MOO+OOO compat is live.

### Deferred for revisit

1. **Reborn Configs MOO INI cleanup** — remove the values that get overwritten anyway, to avoid confusion. Cosmetic, not load-bearing.
2. **Custom Reborn compat INI** — if first playtest shows MOO's flavor is lost (user misses dynamic loot containers), ship `Reborn Configs/Maskar's Oblivion Overhaul for compatibility.ini` that keeps a few MOO systems alongside OOO. Risk: double-loot, double-damage tuning.

## Risks and unknowns

- **APW CR drift vs OOO 1.7**: APW was built against an older OOO. Vampire NPC_ records were renamed in 1.7; boss SCPTs were re-enabled with new spells; clutter prices changed. The analyzer's `target_record_in_current_ooo` column flags this per-record, so we have a knob, but it adds review burden in Phase E. **Estimated impact: tens of records**, not hundreds; SCPT and NPC_ classes most affected.
- **Bashed Patch Leveled Lists patcher**: previously crashed on UOP-no-overhaul. Adding OOO satisfies the precondition, but other crash modes are possible. Phase C's rollback path is documented.
- **OOOExtended decision reversal**: if we later want OOOExtended, the conflict sweep needs re-running and the CR ESP rebuilt — adds maybe one analyzer run. Not blocking.
- **Reborn-OOO playtest duration**: undefined ("an agreed window"). Without a fixed gate, "promote" risks being indefinitely deferred. Suggest a target like "one main quest milestone or 4 in-game hours, whichever first" and document the call.
- **Wabbajack compile size**: OOO BSA (657 MB) + voice files + Shivering Isles BSA (218 MB compressed) materially grow `Reborn-Base.wabbajack` from ~247 MB to ~1.5 GB. Confirm GitHub Release file size limits don't bite.
